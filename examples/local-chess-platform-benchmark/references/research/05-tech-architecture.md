# 维度五：技术架构推测

## 1. 客户端框架

### 微乐客户端技术栈推测

**引擎层：Cocos Creator**
- 微信小游戏生态的主流引擎，微乐大概率使用Cocos Creator 2.x或3.x
- 通过微信小游戏Runtime运行，底层由C++实现的轻量级系统调用层+V8引擎
- 渲染层使用WebGL封装，性能接近原生

来源: [Cocos Runtime技术架构](https://www.nxrte.com/jishu/40794.html)
来源: [Cocos Creator微信小游戏发布](https://docs.cocos.com/creator/3.8/manual/zh/editor/publish/publish-wechatgame.html)

**多玩法框架设计（推测）**
```
┌─────────────────────────────────────────────┐
│              玩法配置层                       │
│   麻将规则JSON    扑克规则JSON    棋类规则JSON │
├─────────────────────────────────────────────┤
│              玩法引擎层                       │
│   麻将引擎(胡牌/听牌/番种)                    │
│   扑克引擎(出牌/比牌/叫分)                    │
│   通用规则解释器                              │
├─────────────────────────────────────────────┤
│              框架层                           │
│   牌桌管理 | 手牌系统 | 动画队列 | 状态机     │
├─────────────────────────────────────────────┤
```

**小程序分包策略（推测）**
- 主包：大厅UI、网络通信、公共框架（<2MB限制）
- 分包：各地方玩法（按需下载，单个分包<2MB）
- 远程资源：美术资源、音效通过CDN按需加载

### 热更新机制
- **Lua热更新**：通过xLua或类似方案，运行时动态加载Lua脚本实现逻辑热更新
- **配置热更新**：规则配置JSON通过服务端下发，客户端缓存+版本校验
- **资源热更新**：Cocos Creator Asset Bundle机制，差量更新美术资源

## 2. 服务端微服务架构

### 棋牌平台服务端通用架构

基于业界实践，棋牌平台服务端通常采用以下架构：

**三层结构**
```
Libraries层 → Core层 → Applications层
```

**Core层核心服务**
| 服务 | 职责 |
|------|------|
| AsyncService | 封装异步调度，基于线程池 |
| TCPServerService | 管理客户端连接池，处理Socket读写 |
| TimerService | 定时器功能与回调 |
| DatabaseService | 数据库连接池，异步DB操作 |

**Applications层服务器类型**
| 服务器 | 职责 |
|--------|------|
| CenterServer | 维护游戏列表和房间信息（游戏类型/站点/房间） |
| LogonServer | 玩家注册和登录认证 |
| LogServer | 游戏日志存储和回放 |
| RoomServer | 管理游戏房间，转发游戏逻辑消息 |

来源: [棋牌游戏服务器架构总体设计](https://cloud.tencent.com/developer/article/1071836)
来源: [棋牌游戏服务器架构详细设计](https://cloud.tencent.com/developer/article/1071639)

### 关键设计决策

**无分区世界服**
- 棋牌游戏不分区不分服，按世界服思想设计
- 服务器是多台物理机的集群
- 使用Redis共享数据，保证用户登录任意服务器都能获取数据

**数据持久化策略**
- 棋牌类游戏数据少、计算量小
- Redis缓存所有用户在线数据
- 异步更新数据库（MySQL/MongoDB）
- 牌局记录写入日志服务，支持回放

**通信协议**
- Protocol Buffer序列化
- 每个通信包含：nMainCmd（请求类别）+ nSubCmd（具体操作）+ nDataSize + pData
- WebSocket长连接为主，HTTP短连接为辅（非实时操作）

来源: [棋牌服务器架构设计-博客园](https://www.cnblogs.com/panchanggui/p/9518825.html)

### 游戏逻辑服务的多玩法支持

**策略模式**
- 定义统一的IGameLogic接口
- 每种玩法实现该接口：MahjongLogic、DouDiZhuLogic、GuanDanLogic
- RoomServer根据房间类型加载对应的游戏逻辑实例

**配置驱动**
- 麻将类玩法差异大但骨架相似，用配置表描述差异点
- 配置项包括：初始手牌数、可用花色、胡牌条件、特殊番种
- 新增地方玩法 = 新配置表 + 少量定制逻辑

## 3. 基础设施推测

### 微乐（腾讯系）
- 云平台：腾讯云
- 容器化：TKE（腾讯容器服务）
- 消息队列：腾讯云TDMQ/Kafka
- 缓存：腾讯云Redis
- 数据库：腾讯云MySQL/TDSQL
- CDN：腾讯云CDN
- 监控：蓝鲸/Prometheus

### JJ竞技（独立部署）
- 自建IDC + 云混合部署
- 自研中间件
- 赛事调度系统自研

### 途游（独立厂商）
- 公有云为主（阿里云/AWS）
- 海外业务需多区域部署
