## 三、游戏逻辑引擎设计

### 规则引擎核心接口

```go
type GameEngine interface {
    Init(config GameConfig) *GameState
    GetLegalActions(state *GameState, playerID string) []Action
    ApplyAction(state *GameState, action Action) (*GameState, error)
    IsTerminal(state *GameState) bool
    GetResult(state *GameState) *GameResult
}
```

### 三个设计原则

1. **不可变状态**：每次操作返回新State，便于回放和并发
2. **双重校验**：客户端预校验 + 服务端最终校验
3. **信息最小化**：只向玩家发送其可见的信息（手牌不能发给对手）

### 状态同步方案

棋牌推荐**状态同步（State Sync）**：
- 服务端权威计算，杜绝作弊
- 每次操作后广播定制化视图（每个玩家看到不同内容）
- 回合制天然适合，延迟要求低（< 500ms可接受）

### 状态机模式（以德州扑克为例）

```
等待入座 -> 准备 -> 发手牌 -> 翻牌前下注
  -> 发翻牌 -> 翻牌后下注 -> 发转牌 -> 转牌后下注
  -> 发河牌 -> 河牌后下注 -> 摊牌/结算 -> 下一局
```

## 六、安全清单（必读）

### 不可妥协的安全底线

| 项目 | 要求 | 错误做法 |
|------|------|---------|
| **洗牌随机数** | crypto/rand（CSPRNG） | math/rand、Math.random() |
| **通信加密** | wss://（WebSocket over TLS） | ws://明文传输 |
| **SQL防注入** | 参数化查询 | 字符串拼接SQL |
| **输入校验** | 服务端校验所有输入 | 信任客户端数据 |
| **操作幂等** | 消息序号+去重 | 允许重复提交 |
| **金额操作** | 整数（分）+事务+日志 | 浮点数+无日志 |

### 经典漏洞案例

**1999年ASF Software在线扑克被攻破**：32位PRNG + 时钟种子 -> 攻击者可计算所有玩家手牌。
教训：随机数安全是棋牌安全的基石，使用CSPRNG是不可妥协的底线。

## 启发式规则（Heuristics）

1. **随机数安全是棋牌安全的基石**——必须使用CSPRNG，这是不可妥协的底线
2. **客户端永远不可信**——所有操作服务端校验，客户端只做预校验和展示
3. **信息最小化**——只发送玩家应该看到的数据，绝不多发（手牌不能发给对手端）
4. **经济系统零容错**——金额操作必须原子、可审计、可追溯
5. **选框架看维护而非星数**——5.5k星但停更5年的Leaf不如2.8k星持续维护的Pitaya
6. **AGPL是商业雷区**——ioGame的AGPL-3.0要求网络服务也必须开源，商用前必须评估
7. **先单体后微服务**——不要过早分布式，单体架构足以支撑大多数早期项目（< 5000 CCU）
8. **压测要模拟真实行为**——不是发N个空消息，而是模拟完整的登录-匹配-对局-结算流程
9. **日志是最好的调试工具**——完整操作日志可回放任何争议牌局
10. **合规第一技术第二**——在搞清楚法律边界之前不要写一行代码（虚拟货币/真金/牌照）
