## 一、创始人背景真实性验证体系

### 1.1 多维度交叉验证框架

| 验证维度 | 方法 | 工具/来源 | 权重 |
|---------|------|----------|------|
| **职业背景** | LinkedIn 履历 + 前同事确认 | LinkedIn, 行业人脉网络 | 高 |
| **链上历史** | 早期钱包活动、协议交互记录 | Arkham Intelligence, Etherscan | 高 |
| **代码贡献** | GitHub commit 历史、PR 质量、持续性 | GitHub, GitLab | 高 |
| **学术/教育** | 学历学位验证 | 学校官网、论文数据库 | 中 |
| **公开记录** | 法庭记录、公司注册、破产记录 | PACER (美国), Companies House (英国) | 高 |
| **媒体足迹** | 会议演讲、播客、采访记录 | YouTube, 行业会议记录 | 中 |
| **社交媒体** | Twitter/X 发帖历史、观点一致性 | Twitter 高级搜索, Wayback Machine | 中 |

### 1.2 关键验证步骤

1. **履历声明交叉验证**：Celsius CEO Alex Mashinsky 声称"拥有30亿美元退出记录"和"2004年最成功的IPO"，均被调查者发现为虚假。任何无法第三方独立验证的履历声明都应标记为疑问。
2. **GitHub 深度审查**：
   - 查看 commit 历史的连续性（融资前突然活跃后沉寂是红旗）
   - 审查 PR 质量，区分"实质贡献"与"修改 README"
   - 检查是否有多个开发者真实参与，还是一人独撑
3. **链上身份验证**：
   - 使用 Arkham Intelligence 关联钱包地址与真实身份
   - 追踪创始人早期链上活动，评估其在行业中的真实经验深度
   - 检查是否有与已知欺诈项目关联的地址

## 二、危险信号模式（Behavioral Red Flags）

### 2.1 Do Kwon 模式——自大型创始人

**核心特征**：
- 公开嘲讽批评者（Do Kwon 在 Twitter 上持续攻击质疑 UST 稳定性的人）
- 声称系统"已经过压力测试并证明了弹性"，但实际未经真正考验
- 内部团队多次提出风险预警被忽视
- 面对监管时采取对抗态度（Kwon 威胁要起诉 SEC）
- 崩盘前的内部人抛售：据报道 Do Kwon 在崩盘前每月套现 8000 万美元
- 崩盘前数天解散公司实体（2022年4月30日解散韩国实体，5月初崩盘）

**教训**：当创始人的公开形象从"自信"越过界限到"自大"，且无法容忍任何质疑时，这不是"强领导力"，而是危险信号。

### 2.2 SBF 模式——道德包装型创始人

**核心特征**：
- 以"有效利他主义"为叙事包装，目标是"赚尽可能多的钱来支持崇高事业"
- 公司治理严重缺失：没有独立董事会、没有合格审计、没有适当的财务簿记
- 使用即时通讯处理发票，用小企业会计软件管理数十亿美元
- 关联实体（FTX 与 Alameda）声称独立但实际深度纠缠
- 投资者尽调失败：Sequoia 在 SBF 边打游戏边开会后反而追加投资
- 早期投资者发现 Alameda 的利润每月都在下降，但"混乱的电子表格"取代了正式审计

**教训**：道德叙事不能替代公司治理。越是强调"使命"而回避透明度的创始人，越需要审视。

### 2.3 Mashinsky 模式——"银行替代者"型创始人

**核心特征**：
- 反复公开攻击传统金融，声称自家产品"风险远低于银行"
- 提供不可持续的高收益（Celsius 提供高达 17% 收益率）
- 在崩盘前一天还在公开挑战质疑者："找出一个从 Celsius 提款有问题的人"——次日即暂停提款
- 私下出售数千万美元个人持仓，同时公开声称项目安全

**教训**：当创始人的公开声明与链上行为严重不符时，链上数据永远比嘴上说的更可信。

## 三、连续创业记录评估

### 3.1 评估框架

| 评估项 | 绿灯信号 | 红灯信号 |
|-------|---------|---------|
| **前项目结局** | 正常退出/收购/过渡 | Rug pull / 不明原因关闭 / 用户资金损失 |
| **社区评价** | 前项目社区仍有正面评价 | 前社区充满控诉和被骗声明 |
| **时间间隔** | 充分复盘后再启动新项目 | 上个项目刚崩盘就立即启新项目 |
| **披露态度** | 主动披露前项目经历和教训 | 隐瞒或淡化前项目历史 |
| **法律状态** | 无未决诉讼或监管行动 | 有进行中的法律纠纷 |

### 3.2 经典案例：0xSifu / Wonderland

**事件**：2022年1月，链上调查者 ZachXBT 揭露 Avalanche DeFi 协议 Wonderland 的联合创始人"0xSifu"的真实身份是 Michael Patryn（原名 Omar Dhanani）——已倒闭加密交易所 QuadrigaCX 的联合创始人。

**Patryn 的犯罪历史**：
- 2005年因身份盗窃、银行和信用卡欺诈被判 18 个月
- 运营过 shadowcrew.com（被盗信用卡交易市场）
- 2007年承认入室盗窃和计算机欺诈
- 两次合法更名以掩盖身份

**后果**：Wonderland 代币 TIME 从历史高点暴跌 97%。创始人 Daniele Sestagalli 承认早就知道 Sifu 的身份，但认为"过去不能决定未来"——这一判断直接导致了投资者的巨额损失。

**教训**：连续创业记录审查必须包括身份验证——伪名不等于匿名，有犯罪历史的人会更换身份重新进入行业。

### 3.3 连续 Rug Pull 模式

链上分析发现，BSC（BNB Chain）上存在"连续 rug pull"操作者，单个诈骗者在 2022 年中期部署了数十个 rug pull 代币，累计获利超过 200 万美元。被称为"字典诈骗者"的连续欺诈者在以太坊、BNB Chain 和 Polygon 上部署了超过 9,000 个诈骗代币。

**识别方法**：
- 使用 Arkham/Nansen 追踪部署者钱包的历史活动
- 检查合约代码模式是否与已知诈骗合约相似
- 审查钱包资金来源链条

## 四、匿名团队判断标准

### 4.1 匿名 vs 伪名：关键区别

| | 伪名（Pseudonymous） | 匿名（Anonymous） |
|---|---|---|
| **定义** | 使用固定假名，有持续的链上/社区记录 | 完全无可追踪身份 |
| **代表** | Satoshi, 许多知名 DeFi 开发者 | 新项目的不明身份团队 |
| **可验证性** | 通过链上历史和社区声誉可评估 | 无法验证任何声明 |

### 4.2 匿名合理的场景（绿灯/黄灯）

- **纯技术协议**：不涉及用户托管资金，代码开源且经审计
- **DAO 治理型项目**：权力真正分散，单人无法控制资金
- **有长期伪名记录**：开发者有多年的链上活动和社区声誉
- **代码即保障**：合约经过多次独立审计，admin key 已销毁或由多签控制
- **监管避险**：在法律环境不明确的地区，匿名保护个人安全

### 4.3 匿名是红旗的场景（红灯）

- **团队可以单方面控制资金**：匿名 + 管理员可暂停合约/铸造无限代币/提取流动性
- **没有可验证的链上历史**：匿名且无法提供任何前项目证据
- **涉及资金托管**：用户需要将资金存入团队控制的合约
- **无时间锁/多签保护**：流动性未锁定，或锁定有到期日（"定时 rug pull"）
- **代币已发行但无产品**：匿名团队在没有工作产品的情况下发行代币
- **拒绝 KYC**：即使是对投资者的私下 KYC 也拒绝
- **统计事实**：DappRadar 2025 年数据显示 92% 的成功 rug pull 来自假名背后的开发者

### 4.4 最佳实践判断流程

```
团队匿名？
  ├── 是否有长期伪名声誉？
  │     ├── 是 → 检查链上历史和社区评价 → 可能可接受（黄灯）
  │     └── 否 → 高风险（红灯）
  ├── 合约是否有管理员权限？
  │     ├── 是 → admin key 是否由多签/时间锁保护？
  │     │     ├── 是 → 可接受（黄灯）
  │     │     └── 否 → 极高风险（红灯）
  │     └── 否（不可变合约）→ 风险较低
  └── 是否涉及用户资金托管？
        ├── 是 → 匿名+托管 = 红灯
        └── 否 → 可接受度更高
```

## 五、团队代币持仓和出售行为追踪

### 5.1 工具矩阵

| 工具 | 核心能力 | 适用场景 | 费用 |
|------|---------|---------|------|
| **Arkham Intelligence** | 地址去匿名化、实体关联、资金流可视化 | 识别团队钱包真实身份、追踪跨链资金流 | 个人免费 |
| **Nansen** | 5亿+标记钱包、Smart Money 追踪、Token God Mode | 分析代币分布、持仓者行为、鲸鱼动向 | 付费 |
| **Tokenomist.ai** | 代币解锁日历、归属时间表 | 追踪即将到来的团队/投资者解锁事件 | 免费/付费 |
| **DropsTab** | 归属进度可视化、解锁规模分析 | 评估解锁对市场的潜在冲击 | 免费/付费 |
| **Etherscan/Solscan** | 基础链上数据查询 | 验证具体地址活动 | 免费 |

### 5.2 关键监控指标

1. **解锁规模 vs 流通供应**：解锁量占流通供应 >5% 时需要特别关注
2. **接收者类型**：团队和早期投资者的解锁历史上是"最易卖出"的类别
3. **吸收比率**：当前流动性是否足以消化解锁量而不引起重大价格波动
4. **Cliff 解锁模式**：大额一次性解锁的典型价格行为——解锁前约30天开始走弱，最后一周加速抛售，解锁后10-14天逐步稳定
5. **低流通/高FDV 项目**：忽视代币解锁会显著增加散户成为风投退出流动性的风险

### 5.3 团队抛售红旗清单

- 团队钱包在公告利好前/后大量转移代币至交易所
- 团队成员通过多层中间钱包转移代币试图混淆追踪
- 代币解锁后立即出现大额卖单
- 团队持仓比例远超白皮书声明
- 创始人口头声称"长期持有"但链上数据显示持续减仓
- 使用 Tornado Cash 等混币器转移团队代币

### 6.2 ZachXBT 调查方法论总结

ZachXBT 是独立区块链调查者，已帮助受害者追回超过 3.5 亿美元。其核心方法论：

1. **区块链取证**：地址聚类分析，识别同一人/组织控制的多个钱包
2. **资金流追踪**：跨链、跨交易所的完整资金流向图
3. **OSINT 交叉验证**：将链上数据与域名注册、法庭记录、社交媒体活动交叉比对
4. **社区情报网络**：监控 Telegram 和 Discord 中的犯罪分子聚集群
5. **模式识别**：识别跨项目的重复欺诈模式和地址关联
6. **公开发布**：通过 Twitter 线程和长文报告实时公开调查结果

- rug pull 仍是最常见犯罪类型：每年超过 30 万个诈骗代币被创建，200 万投资者被骗

## 七、交通灯判断标准

### 7.1 绿灯（低风险）

- [ ] 创始团队身份公开且可通过多个独立来源验证
- [ ] 团队成员有可追溯的行业经验和成功项目记录
- [ ] GitHub 有持续、实质性的代码贡献历史
- [ ] 代币归属计划合理（团队份额 >= 2 年归属期 + 6 月 cliff）
- [ ] 团队代币持仓行为与公开声明一致（链上可验证）
- [ ] 有知名机构/投资者背书且这些机构做过真正的尽调
- [ ] 公司治理结构完善：独立董事、合规审计、透明财务
- [ ] 创始人能够建设性地回应批评和质疑
- [ ] 前项目有正面结局或透明的经验教训总结

### 7.2 黄灯（需进一步调查）

- [ ] 团队为伪名但有较长链上声誉记录
- [ ] 创始人来自传统行业，加密行业经验有限
- [ ] 团队代币归属期偏短（1-2 年）或 cliff 偏短
- [ ] 部分团队成员信息难以完全验证
- [ ] 前项目结局不明确但无明显恶意行为
- [ ] 创始人偶有夸大但未到系统性虚假的程度
- [ ] 投资者阵容中有知名名字但投资金额不明
- [ ] 匿名团队但合约不可变且经过审计

### 7.3 红灯（高风险/建议否决）

- [ ] 创始人身份无法验证或履历存在可证伪的虚假声明
- [ ] 团队成员有 rug pull、欺诈、或重大法律纠纷历史
- [ ] 匿名团队 + 合约有管理员权限 + 无多签/时间锁保护
- [ ] 团队代币无归属期或归属期极短（<6 个月）
- [ ] 链上数据显示团队正在大量抛售但公开声称"长期持有"
- [ ] 承诺不可持续的高收益率（> 20% APY 且无法解释来源）
- [ ] 创始人攻击/嘲讽所有批评者，无法接受任何质疑
- [ ] 关联实体间存在不透明的资金流动
- [ ] 拒绝任何形式的审计或透明度要求
- [ ] 公司实体在监管不透明的离岸司法管辖区注册且无其他实体

## 八、快速否决清单（Team Kill List）

以下任一条件满足即可立即否决，无需进一步分析：

| # | 否决条件 | 参考案例 |
|---|---------|---------|
| 1 | 创始人有已证实的 rug pull 或金融欺诈前科 | 0xSifu/QuadrigaCX |
