## 一、通胀型积分经济模型（核心框架1）

### DeFi代币 vs 休闲积分的本质差异

| 维度 | DeFi代币 | 休闲游戏积分 |
|------|---------|------------|
| 供给模型 | 有限/通缩 | 无限/通胀 |
| 设计目标 | 升值、投资回报 | 保持游戏流动性 |
| 可交易性 | 自由交易 | 合约级禁止（Soulbound） |
| 用户心理 | 囤积、投机 | 消耗、娱乐 |
| 新玩家门槛 | 需购买代币 | 登录即送 |

### Faucet/Sink双循环

**Faucet（产出）**：每日登录工资（余额<阈值时触发）、对局奖励、活动发放、新手礼包、任务系统

**Sink（消耗）**：对局入场费（40-50%）、道具购买（20-30%）、高级场门槛（15-20%）、系统税（5-10%）、外观装饰（5-10%）

### 通胀率控制

```
目标：周通胀率 3-8%
杠杆：日工资发放量 = f(DAU, 平均余额, 目标通胀率)
- 实际通胀 > 目标 → 提高入场费 / 推出限时高消耗活动
- 实际通胀 < 目标 → 加大日工资 / 降低入场费
```

### 技术实现

推荐方案：ERC-20 + `_beforeTokenTransfer` override——只允许mint（from=0）和burn（to=0），禁止用户间转账。兼容性好，工具链完善。不用SBT（ERC-5192），因为积分是同质化的，NFT标准表达同质化积分是语义错误。

## 二、链上链下三层数据架构（核心框架2）

### 数据分层模型

| 层级 | 内容 | 上链理由 |
|------|------|---------|
| Layer 1: 链上 | 积分总余额快照、大额mint/burn、跨游戏转移、治理参数变更 | 不可篡改，用户可自证 |
| Layer 2: 链下+证明 | 对局结果摘要（Merkle Root上链）、批量积分变动、道具购买 | 可验证但不需实时上链 |
| Layer 3: 纯链下 | 实时对局过程、出牌记录、匹配队列、临时Buff | 高频低价值，丢失可恢复 |

### 上链判断五问

1. 用户是否需要自证拥有？→ 是则上链
2. 跨游戏/跨平台是否需要读取？→ 是则上链
3. 是否涉及资产所有权变更？→ 是则上链
4. 频率是否超过1次/秒？→ 是则链下
5. 数据丢失是否可恢复？→ 不可恢复则上链

### Gas优化四策略

1. **批量结算**：累积100局或5分钟一次性上链，Gas节省95%
2. **Merkle Tree证明**：每日全服余额Root Hash上链（约45000 gas），用户通过Proof自证
3. **L2选型**：推荐Base（合规友好，<$0.01/tx）或自建OP Stack App Chain（完全控制Gas策略）
4. **Paymaster代付**：ERC-4337 Paymaster为玩家代付Gas，每用户每月$0.01-0.05

## 三、钱包无感接入方案（核心框架3）

### 核心原则：钱包必须"隐形"

休闲棋牌玩家画像为非加密原住民。要求安装MetaMask或记助记词，99%用户直接流失。

### 推荐组合：MPC + AA

```
用户视角：点击"微信登录"/"手机号登录" → 游戏直接开始
后台：OAuth → MPC密钥分片生成 → ERC-4337 Smart Account → Paymaster注册
用户全程不知道自己有个"链上地址"
```

### 服务商选型

| 服务商 | 优势 | 适用场景 |
|--------|------|---------|
| Particle Network | Unity/Unreal引擎原生支持最好 | 游戏引擎深度集成 |
| Web3Auth | 生态最大（MetaMask收购），73%新项目使用ERC-4337 | Web优先项目 |
| Privy | Stripe收购，内置Paymaster | React生态 |
| Openfort | 游戏专用 | 需要游戏特化功能 |

### Session Key优化

一次授权，限时自动签名（24小时有效期）。权限范围限于积分mint/burn和小额道具购买，大额转移需重新授权。避免每次链上操作弹出确认框打断游戏。

## 四、跨游戏积分互通架构（核心框架4）

### 单一代币 + 游戏侧参数调控（推荐）

所有游戏共享一个HappyPoints合约，链上只管mint/burn/balance。每个游戏独立设置入场费、奖励、道具价格的乘数因子，参数调控在链下完成。

### 购买力平价（PPP）设计

核心原则：**1小时游戏消耗的净积分应相近**

```
净消耗/小时 = (入场费 × 局数/h) - (平均奖励 × 局数/h)
示例目标：所有游戏净消耗 ≈ 120积分/小时
斗地主：50×12 - 40×12 = 120/h
麻将：  200×3 - 160×3 = 120/h
德州：  300×2 - 240×2 = 120/h
```

### 动态调节

监控各游戏DAU占比、净积分流出/流入、跨游戏用户消耗分布。若某游戏积分净流出过大则降低入场费或提高奖励，反之亦然，保持各游戏积分池不出现单向大幅流动。

### 体验对齐原则

用户层面"感觉跟QQ欢乐豆一模一样"：注册送豆、日工资、打牌输赢、买道具、余额查询——所有操作的用户感知差异为零。

### 绝不暴露的Web3细节

钱包地址、Gas费、区块确认等待、交易Hash、链名/网络切换、助记词/私钥——全部对用户隐藏。

### Web3带来的5项增量价值

1. **可验证公平性**：每局Merkle Root上链 + VRF随机数，用户可独立验证"系统没有作弊"
2. **数据所有权**：积分在链上，即使游戏关服数据仍在，封号无法清除链上余额
3. **跨平台便携性**：积分跟随钱包地址，任何授权游戏均可读取，"游戏资历"可携带
4. **防灰色交易**：合约级禁止P2P转账从技术上消灭豆商，所有mint可追溯
5. **社区治理**（进阶）：活跃用户DAO投票决定日工资量、活动奖励、道具定价

## 决策启发式

1. **"通胀是Feature"法则**：积分设计的第一问不是"怎么限制供给"，而是"怎么让新玩家永远有豆玩"。如果你在考虑通缩机制，说明方向错了。

2. **"上链五问"筛选法**：任何数据决定是否上链前，过一遍五问清单（自证需求、跨平台需求、所有权变更、频率、可恢复性）。五问全否则绝不上链。

3. **"1小时等值消耗"定价锚**：跨游戏定价不看单局成本，看每小时净消耗。所有游戏的净消耗/小时应在同一数量级，否则积分会单向流动。

4. **"妈妈测试"体验标准**：如果你妈妈（非加密用户）注册后10秒内不能开始玩游戏，钱包方案就不合格。钱包地址、Gas、助记词——任何Web3术语出现在普通用户界面都是设计失败。

5. **"三条红线"合规底线**：不暗示投资回报、不开放二级市场、不承诺回购。只要守住这三条，不可交易积分在大多数司法管辖区不会被定性为证券。

6. **"先单链后跨链"架构原则**：初期所有游戏部署在同一L2（如Base），避免跨链复杂性。跨链桥接引入安全风险，等生态成熟后再考虑CCIP。

7. **"Batch优先"Gas策略**：默认使用批量结算（100局或5分钟一批），单笔上链只用于大额事件。95%的Gas节省来自这一条。

8. **"链下缓存即时响应"原则**：用户查余额先读链下缓存，需要证明时才查链上。区块确认延迟绝不暴露给用户，链下即时响应是底线。

9. **"中国大陆双轨制"策略**：大陆版本纯链下运行，海外版本上链。技术架构从一开始就预留链上/链下切换能力，通过配置决定数据走向。

10. **"隐形上链先行"节奏**：Phase 1必须做到用户完全无感知。任何Web3功能暴露都应是可选的、渐进的。先证明技术可行且不损失用户体验，再逐步释放Web3价值。

## 诚实边界

1. **L2生态仍在快速变化**：Base、Immutable zkEVM等L2的费用结构和稳定性仍在演进，选型建议基于当前状态，半年后可能需要重新评估。

2. **嵌入式钱包服务商存在平台风险**：Web3Auth被MetaMask收购、Privy被Stripe收购——服务商整合可能导致API变更或定价调整，建议保持抽象层以便切换。

3. **合规判例尚不充分**：链上不可交易积分是较新的法律形态，多数司法管辖区缺乏明确判例。特别是韩国和印度的定性仍有不确定性，正式运营前应获取当地法律意见。

4. **跨游戏PPP平衡是持续运营问题**：购买力平价公式提供起点，但实际平衡需要根据真实玩家行为数据持续调优，不存在一劳永逸的参数设定。

5. **DAO治理在游戏场景未经大规模验证**：Phase 4的社区治理是理论上的增量价值，实际中游戏经济参数的治理可能面临投票率低、恶意提案等问题，需谨慎推进。
