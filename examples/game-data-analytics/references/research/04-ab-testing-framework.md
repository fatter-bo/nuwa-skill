# 研究维度四：A/B测试框架

## 1. A/B测试全流程框架

### 六步法：假设→指标→分组→样本量→时长→分析

**Step 1：提出假设**
- 基于数据或观察提出可验证假设
- 格式：「如果改变X，则指标Y会提升Z%，因为原因W」
- 示例：「如果将新手引导从7步缩短为4步，则D1留存会提升5%，因为当前Step 5流失率达40%」
- 关键：假设必须明确、可测量、有商业价值

**Step 2：选定指标**
- **主指标（Primary Metric）**：假设直接影响的指标（如D1留存率）
- **护栏指标（Guardrail Metric）**：确保改动不伤害其他方面（如ARPDAU、Session时长）
- **诊断指标（Diagnostic Metric）**：帮助理解"为什么"（如教程完成率、首局完成率）
- 避免选择过多主指标（建议1个主+2-3个护栏）

**Step 3：用户分组**
- **随机分组**：确保Control和Treatment组在人口统计和行为特征上均衡
- **避免偏差**：新用户实验最干净（无历史行为干扰）；老用户需做AA测试验证分组均衡性
- **分流比例**：通常50/50（统计效能最高），风险较大的改动可90/10或95/5

**Step 4：计算最小样本量**
- 四个输入参数：
  - **基础转化率（Baseline）**：当前指标值
  - **最小可检测效应（MDE）**：希望检测到的最小变化幅度（通常2-5%）
  - **显著性水平（α）**：通常0.05（95%置信度）
  - **统计功效（1-β）**：通常0.80（80%）
- **经验公式**：MDE越小→需要样本量越大；基础转化率越接近50%→需要样本量越大
- 一般规则：MDE 2-5%是合理范围；没有历史数据时插入2-5%的MDE
- 工具推荐：Optimizely Sample Size Calculator、Statsig Calculator

**Step 5：确定测试时长**
- 最少覆盖一个完整周周期（7天），以消除日间差异
- 游戏A/B测试通常需要至少2周，以排除新奇效应和周期性行为
- 考虑因素：日活量（DAU越大→所需天数越少）、季节性事件（避免活动期间干扰）
- 提前确定结束日期，**不要**根据中间结果提前结束

**Step 6：分析与决策**
- 确认统计显著性达标（p < 0.05）
- 检查效应方向和大小是否符合预期
- 检查护栏指标是否受损
- 分维度查看结果（按地区/新老/设备等），警惕Simpson悖论
- 记录结论和学习，无论结果正负

**来源**：SuperScale "A/B Testing in Mobile Games"; Databricks "Building an A/B Testing Analysis Framework for Mobile Gaming"; Optimizely Sample Size Calculator
**可信度**：高 — 行业实践与统计学方法论结合

## 2. 最小样本量计算

### 核心公式

```
n = (Z_α/2 + Z_β)² × (p₁(1-p₁) + p₂(1-p₂)) / (p₁ - p₂)²
```

其中：
- Z_α/2 = 1.96（α=0.05时）
- Z_β = 0.84（β=0.20，即Power=80%时）
- p₁ = 基础转化率
- p₂ = 期望转化率（p₁ + MDE）

### 速查表

| 基础转化率 | MDE | 所需每组样本量（α=0.05, Power=80%） |
|-----------|-----|-------------------------------------|
| 5% | 1pp | ~4,700 |
| 5% | 2pp | ~1,300 |
| 30% | 3pp | ~2,800 |
| 30% | 5pp | ~1,000 |
| 50% | 5pp | ~780 |

### 实操建议

- DAU 10K的游戏：50/50分流，每组5K/天，7天=35K样本——足以检测留存率3pp的MDE
- DAU 1K的小游戏：需要跑更长时间或接受更大MDE（5%+）
- 付费率测试（基础2-3%）：需要极大样本量，建议选择ARPDAU作为替代指标

## 3. 常见测试场景

### 按游戏系统分类

| 测试类别 | 典型场景 | 主指标 | 护栏指标 |
|---------|---------|--------|---------|
| **FTUE优化** | 教程步骤数/跳过选项/首局难度 | D1留存 | Session时长 |
| **经济系统** | 首充价格/礼包内容/折扣幅度 | 付费率/ARPU | D7留存 |
| **广告变现** | 广告频次/插屏时机/激励视频奖励量 | eCPM/广告收入 | D7留存/Session时长 |
| **社交功能** | 好友推荐时机/排行榜展示/分享奖励 | K-Factor/D7留存 | ARPDAU |
| **LiveOps** | 活动类型/奖励力度/活动时长 | DAU/Session次数 | ARPDAU |
| **UI/UX** | 按钮位置/色彩方案/弹窗时机 | 转化率/点击率 | D1留存 |

### 棋牌游戏特有测试场景

- **匹配算法**：ELO匹配 vs 宽松匹配 → 对局等待时间 vs 胜率分布
- **Bot策略**：Bot占比30% vs 50% → 匹配速度 vs 用户感知
- **等级场设计**：进入门槛/奖励倍率 → 场均局数 vs 等级场迁移率
- **筹码经济**：每日赠送量/破产补助 → Session数 vs 付费转化

## 4. 典型陷阱与规避

### 陷阱一：窥探效应（Peeking Problem）

**定义**：在实验期间提前查看数据，并基于中间结果做出提前终止决策。

**危害**：越频繁查看数据，假阳性率越高。如果每天"偷看"一次，7天后假阳性率可能从5%膨胀到20%+。

**规避方法**：
- 预先确定结束日期和样本量，严格执行
- 如果需要提前停止的灵活性，使用Sequential Testing方法（如CUPED或Bayesian方法）
- 设置自动化报告，在达到预定样本量前不输出显著性判断

### 陷阱二：辛普森悖论（Simpson's Paradox）

**定义**：在各子群体中呈现一致趋势的数据，在汇总后趋势反转。

**游戏中的典型场景**：
- A版本在新用户中胜出，在老用户中也胜出，但合并后B版本胜出（因为A组老用户比例更高，老用户整体指标更低）
- 不同国家分别看都是正向，合并后变负向（因国家间基数差异大）

**规避方法**：
- 确保分组在关键维度（新/老、国家、设备）上均衡
- 分维度查看结果，不仅看汇总数据
- 使用分层随机化（Stratified Randomization）

### 陷阱三：新奇效应（Novelty Effect）

**定义**：用户因新功能的新鲜感而表现出高于正常水平的参与度，但随时间消退。

**危害**：在新功能测试中产生过度乐观的结论，上线后指标回落。

**规避方法**：
- 测试至少运行2周（让新鲜感消退）
- 分新用户/老用户分析：新用户无"新奇"基线，老用户有
- 观察指标随时间的变化趋势，而非只看平均值
- 如果测试前几天指标飙升后回落，可能是新奇效应

### 陷阱四：多重比较问题

**定义**：同时测试多个变体或多个指标，导致假阳性率膨胀。

**规避方法**：
- 使用Bonferroni校正：显著性阈值 α/k（k为比较次数）
- 区分"主指标"（严格校正）和"探索性指标"（允许宽松标准）
- 减少同时测试的变体数量

### 陷阱五：选择偏差

**定义**：实验组和对照组在分组时就已不均衡。

**规避方法**：
- 做AA测试验证随机分组质量
- 检查分组间的基线指标是否一致
- 使用设备ID或用户ID级别的稳定分流
