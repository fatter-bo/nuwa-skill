## 一、四种定价模型与数据需求

### LMSR（对数市场评分规则）— Robin Hanson发明
**公式**：C(q) = b × log(Σ exp(q_i / b))
- 参数b控制流动性深度，最大亏损 = b × ln(n)
- **数据需求**：仅需当前份额向量q和b参数，不需要外部价格
- **FastEvent关联**：LMSR市场仍需结算——事件结果确认后才能分配赢家。FastEvent提供秒级结算数据
- **使用者**：Augur v1、企业预测市场（Cultivate Labs）

### CLOB订单簿 — Polymarket
- 链下撮合+链上结算，基于条件代币框架（CTF）
- Yes/No代币定价$0-$1.00，完全抵押
- **数据需求**：依赖做市商报价（Susquehanna、Jump Trading），结算需要预言机确认结果
- **FastEvent关联**：Polymarket当前用UMA结算，2h+延迟。FastEvent可作为UMA替代/补充
- **关键指标**：2025年$215亿交易量，2026.02月$70亿+

### vAMM — Azuro
- 不需要引导流动性，从单例流动性池（Liquidity Tree）继承
- 数据提供者设置基准赔率 → 投注量动态调整
- **数据需求**：初始赔率（链下数据源）+ 实时事件数据（Chainlink预言机）+ 结算数据
- **FastEvent关联**：Azuro使用Chainlink+自建预言机混合方案。FastEvent可作为更快、更结构化的体育数据供应商
- **关键指标**：$10亿+累计预测量，4.28% take rate，50+ dApp

### Parimutuel — 传统赛马模式
- 所有投注汇入资金池，扣除抽成后按比例分配给赢家
- 赔率在投注关闭前不确定
- **数据需求**：仅需最终资金池总额+事件结果
- **FastEvent关联**：结算简单但仍需权威事件结果数据

## 二、条件代币框架（Gnosis CTF / ERC-1155）

Polymarket的核心技术基础：

### 市场创建流程
1. 定义三要素：`questionId`（UMA辅助数据的哈希）、`oracle`（UMA CTF Adapter地址）、`outcomeSlotCount`（二元=2）
2. keccak256哈希生成`conditionId`
3. Collection ID = 父集合 + conditionId + indexSet（位掩码：1=Yes, 2=No）
4. Position ID = 抵押代币 + Collection ID → 最终ERC-1155 token ID

### 三个核心操作
| 操作 | 流程 | 作用 |
|------|------|------|
| Split | 锁定1 USDC → 获得1 Yes + 1 No | 创建头寸 |
| Merge | 销毁1 Yes + 1 No → 取回1 USDC | 退出头寸 |
| Redeem | 结算后销毁赢方代币 → 获得$1.00 | 兑现利润 |

**完全抵押不变式**：每对Yes/No代币始终由恰好$1.00支持。

**对FastEvent的意义**：CTF的Redeem操作依赖预言机报告结果。谁能更快、更可靠地报告结果，谁就控制了CTF生态的结算层。

## 六、监管格局

### CFTC分类的历史意义
CFTC将预测市场事件合约归类为**衍生品（swaps）**而非赌博——这使其受联邦金融监管而非州赌博法管辖。

2025年DCM认证了~1,600个事件合约——前所未有的增长。

### FastEvent的监管定位
**FastEvent作为数据层不直接受赌博监管**——这是关键的战略优势：
- 我们提供数据，不运营市场
- 类似Sportradar在传统体育博彩中的定位——数据供应商不是博彩运营商
- CFTC对数据可靠性有要求，但不直接监管数据提供者
- 合规预测市场（Kalshi、Polymarket DCM）反而更需要可审计的数据源——FastEvent的链上记录满足审计要求
