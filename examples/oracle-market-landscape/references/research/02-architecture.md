# 02-architecture: 四种预言机架构模式技术对比

## Push模型（Chainlink DON）

### 工作流程
1. 多个独立节点运营者从多个数据源获取数据
2. DON（Decentralized Oracle Network）聚合（取中位数）
3. 按预设规则推送到链上合约：
   - 心跳（如每1小时）
   - 偏差阈值（如价格变化>0.5%）
4. 数据常驻链上，DApp免费读取

### 技术参数
- Gas成本：每次更新$5-50+（以太坊，取决于gas价格）
- 更新频率：可配置（1秒到1小时）
- 链上存储：是（proxy合约）
- Data Streams产品增加了低延迟能力

### 优势
- 战斗检验8年+，保障$27.4万亿价值
- 数据始终在链上可用——消费者不需要做任何事
- 90%+ DeFi集成
- 最高安全性和机构信任

### 劣势
- Gas成本高（尤其多链部署）
- 心跳间数据可能过时
- 长尾资产不经济——维护21节点DON服务一个小众token不划算
- 不适合一次性事件数据

---

## Pull模型（Pyth Network）

### 工作流程
1. 第一方数据提供者（交易所、做市商）签名价格数据
2. 数据存储在Pythnet（Solana-based appchain）
3. 消费者在需要时拉取数据并提交上链（支付~$0.01微费）
4. 数据只在被消费时才上链

### 技术参数
- Gas成本：~$0.01每次消费（消费者付费）
- 链下延迟：<400ms平均
- 链上新鲜度：取决于何时拉取
- 置信区间：提供价格的置信区间

### 优势
- Gas成本大幅降低
- 第一方数据（直接来自交易所/做市商）——更新鲜
- 亚秒级延迟
- 低成本扩展到多链

### 劣势
- 数据不在链上被动可用——增加消费者复杂度
- MEV风险：消费者提交的更新交易在mempool可见
- 比Push模型更新、战斗检验少
- 第一方数据的信任假设——数据提供者可以作恶

## 乐观模型（UMA Optimistic Oracle）

### 工作流程
1. **请求**：合约发送数据请求（标识符、时间戳、辅助数据、保证金要求）
2. **提案**：提案者提交结果 + $750 USDC保证金
3. **挑战期**：2小时（可配置2h-2天）
   - 无争议 → 结果被接受
   - 有争议 → 争议者提交匹配保证金
4. **DVM投票**：UMA代币持有者commit-reveal投票
   - Commit阶段：24小时
   - Reveal阶段：24小时
   - 法定人数：5M UMA + 65%质押者
5. **结算**：胜方获得保证金，败方失去保证金

### 技术参数
- 无争议延迟：2小时+
- 争议延迟：48-96小时
- 提案保证金：$750 USDC（最低$500）
- 投票法定人数：5M UMA（GAT）+ 65%参与（SPAT）
- MOOV2白名单提案者：37个地址

### 优势
- 可处理任意数据类型（不只是价格）
- 无争议时Gas成本低
- 博弈论安全——需要>50%投票者非理性
- 灵活性——从体育到选举到地缘政治

### 劣势
- 速度慢（小时到天）
- 鲸鱼操纵风险（已发生：乌克兰$700万、UFO$1600万）
- 低投票参与率（只有~15M/100M+参与）
- 4个鲸鱼控制>40%供应——寡头投票
- 主观/模糊问题解决质量差
- MOOV2白名单牺牲了去中心化

## 权威签名模型（FastEvent）

### 工作流程
1. 受信任的权威数据源生成数据
2. 运营方用私钥签名
3. 推送到链上（或提供供拉取）
4. 消费者读取——即时终局

### 技术参数
- 延迟：秒级（仅受出块时间限制）
- Gas成本：单笔交易（最低）
- 信任假设：完全依赖运营方
- 数据签名：可链上验证

### 优势
- 最快解决速度
- 最低成本
- 最简架构——无共识开销
- 适合天然有单一权威源的数据（体育联赛比分、AP通社选举结果）

### 劣势
- 单点故障
- 私钥泄露 = 完全被控制
- 信任依赖运营方诚实和运营能力
- 无冗余

### 信任构建机制
1. 开源代码
2. 多签钱包管理
3. 链上所有结算的可审计记录
4. SLA承诺（可用性和准确性）
5. 多云部署（AWS + GCP + Azure）
6. 透明的数据来源方法论
