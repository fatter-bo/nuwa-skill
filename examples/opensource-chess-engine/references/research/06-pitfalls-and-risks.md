# 06-常见坑和避雷

## 一、安全漏洞

### 1. 随机数可预测

**经典案例**：1999年ASF Software公司的在线扑克被攻破。攻击者发现：
- 洗牌算法使用32位PRNG（伪随机数生成器）
- 种子基于服务器时钟
- 通过与服务器时钟同步，可将可能的牌组从2.08x10^67减少到约200,000种
- 实时穷举搜索后可精确知道所有玩家手牌

来源：https://datamation.com/entdev/article.php/616221

**防范措施**：
- 使用密码学安全随机数生成器（CSPRNG）
  - Go: `crypto/rand`
  - Java: `java.security.SecureRandom`
  - Node.js: `crypto.randomBytes()`
  - Python: `secrets`模块
- **绝对不要**使用`math/rand`、`Math.random()`、`java.util.Random`做洗牌
- 种子不能基于可预测值（时间戳、PID等）
- 64位以上种子空间（32位可被暴力破解）
- Fisher-Yates洗牌算法（而非简单swap）

### 2. 通信未加密

**风险**：
- 明文WebSocket（ws://）可被中间人窃听所有消息
- 玩家手牌、下注金额等敏感信息泄露
- 可被篡改消息（如修改下注金额）

**防范**：
- 必须使用wss://（WebSocket over TLS）
- 服务器配置有效的SSL/TLS证书
- 禁用过时的TLS版本（< TLS 1.2）
- 消息签名防篡改（对关键操作加HMAC）

来源：https://cheatsheetseries.owasp.org/cheatsheets/WebSocket_Security_Cheat_Sheet.html

### 3. SQL注入

**风险**：WebSocket消息中的注入攻击与HTTP一样危险。

**防范**：
- 参数化查询（Prepared Statements），永远不要拼接SQL
- 服务端输入验证和消毒
- ORM框架自带防注入（但要注意原生SQL查询）
- 最小权限原则：游戏服务器数据库账号不应有DROP/ALTER权限

### 4. WebSocket特有攻击

| 攻击类型 | 说明 | 防范 |
|---------|------|------|
| CSWSH（跨站WebSocket劫持） | 利用cookie自动发送 | 验证Origin头 |
| DoS | 大量连接耗尽服务器资源 | 连接速率限制、IP黑名单 |
| 消息注入 | XSS/命令注入 | 输入校验+输出编码 |
| 重放攻击 | 重放历史消息 | 消息序号+时间戳校验 |
| 认证绕过 | 未校验WebSocket升级请求 | Ticket-based认证 |

来源：https://ably.com/topic/websocket-security

### 5. 业务逻辑漏洞

| 漏洞 | 说明 | 防范 |
|------|------|------|
| 重复提交 | 同一操作多次执行 | 操作序号+幂等性校验 |
| 时序攻击 | 利用操作顺序漏洞 | 服务端严格控制操作轮次 |
| 金额篡改 | 客户端发送非法金额 | 服务端独立计算金额 |
| 信息泄露 | 接口返回过多信息 | 最小信息原则 |
| 并发漏洞 | 同时在多个房间扣款 | 分布式锁+事务 |

## 二、协议合规

### 1. 赌博法律风险

**核心区分**：
- 虚拟货币+不可提现 = 一般视为合法游戏
- 涉及真实货币/可提现 = 可能构成赌博
- 各国/地区法规差异巨大

**建议**：
- 明确业务模式是"娱乐棋牌"还是"竞技棋牌"
- 虚拟货币不可提现的设计是合规底线
- 如果涉及真金，必须获取当地牌照
- 接入实名认证和未成年人保护

### 2. 数据隐私

- GDPR/CCPA等数据保护法规
- 用户数据加密存储
- 提供数据删除接口
- 日志中不记录敏感信息

### 3. License合规

| License | 商业使用 | 修改要求 | 传染性 |
|---------|---------|---------|--------|
| MIT | 自由 | 无 | 无 |
| Apache-2.0 | 自由 | 保留声明 | 无 |
| AGPL-3.0 | 需开源 | 必须开源修改 | 强传染 |

**特别注意**：ioGame使用AGPL-3.0，如果基于它开发的服务通过网络提供给用户，必须开源你的全部代码。

## 三、代码质量

### 1. 开源项目常见代码问题

| 问题 | 表现 | 风险 |
|------|------|------|
| 缺少测试 | 0测试覆盖率 | 改一处坏三处 |
| 硬编码配置 | 数据库密码写在代码里 | 安全+部署困难 |
| 无错误处理 | panic/忽略error | 服务不稳定 |
| 全局状态 | 全局变量满天飞 | 并发bug、测试困难 |
| 文档缺失 | 无注释无README | 上手成本高 |
| 依赖过时 | 多年未更新依赖 | 安全漏洞 |

### 2. 评估开源项目质量的检查清单

- [ ] 最近6个月内有commit
- [ ] Issues有人回复/处理
- [ ] 有CI/CD（GitHub Actions等）
- [ ] 测试覆盖率 > 50%
- [ ] README完整（安装/配置/使用/API）
- [ ] 代码有注释
- [ ] 无硬编码密码/密钥
- [ ] 依赖版本不过时
- [ ] License与你的商业模式兼容
- [ ] 性能基准测试

### 3. 二次开发前必做

1. **Fork而非直接修改**：保留上游更新能力
2. **跑通所有测试**：确保基线正常
3. **建立CI**：每次提交自动测试
4. **安全扫描**：用gosec/spotbugs/npm audit扫一遍
5. **性能基线**：改动前先跑压测记录基线数据

## 四、运维踩坑

### 1. 内存泄漏
- Go：goroutine泄漏最常见（忘记关channel、context未cancel）
- Java：对象池未回收、事件监听未注销
- Node.js：闭包引用、EventEmitter未移除

### 2. 状态不一致
- 服务器重启后房间状态丢失
- 方案：定期快照到Redis + 启动时恢复

### 3. 时区问题
- 排行榜重置时间、活动开始结束时间
- 所有服务器统一UTC，客户端本地化显示

### 4. 热更新失败
- 更新代码后旧连接仍用旧逻辑
- 方案：灰度发布 + 旧房间自然结束后下线

## 五、启发式规则（Heuristics）

1. **随机数安全是棋牌安全的基石**：必须使用CSPRNG，这是不可妥协的底线
2. **客户端永远不可信**：所有操作在服务端校验，客户端只做预校验和展示
3. **信息最小化**：只发送玩家应该看到的数据，绝不多发
4. **经济系统零容错**：金额操作必须原子、可审计、可追溯
5. **选框架看维护而非星数**：5.5k星但停更5年的Leaf不如2.8k星持续维护的Pitaya
6. **AGPL是商业雷区**：使用ioGame等AGPL项目前必须评估合规成本
7. **压测要模拟真实行为**：不是"发N个空消息"，而是模拟完整的登录-匹配-对局-结算流程
8. **先单体后微服务**：不要过早分布式，单体架构足以支撑大多数早期项目
9. **日志是最好的调试工具**：完整的操作日志可以回放任何争议牌局
10. **合规第一技术第二**：在搞清楚法律边界之前不要写一行代码
